# Project Workbench Pages Requirements and Implementation Plan

Status: Draft
Target milestone: post-v1.3
Last updated: 2026-06-30

## Summary

Mutsumi's PRES should evolve from a local project launcher into an AI-native project-side data window. The first step is a read-only, schema-driven Project Workbench Page system.

This feature lets each project define one or more custom pages that display project context, local files, Git status, automation placeholders, agent summaries, and future Mutsumi's-SYNC requests. Pages are project-scoped, reviewable as files, and safe to generate with AI because PRES renders a constrained schema instead of executing arbitrary UI code.

## Background

Mutsumi's-SYNC and Mutsumi's PRES have separate responsibilities:

- SYNC is a cross-device coordination and memory service.
- SYNC sends likely user requests to PRES and receives requests from PRES.
- SYNC manages global memory injected into PRES-side AI conversations.
- SYNC receives summarized PRES conversation records.
- SYNC does not own project automation, custom pages, local execution, or agent orchestration.

PRES owns the project-side runtime:

- Project metadata, grouping, editors, templates, Git operations, and local paths.
- Local project context and project-specific UI.
- Future agent calls to specialized tools such as Codex, opencode, text agents, and publishing agents.
- Future automation workflows and project-specific workbench pages.

The long-term direction is a personal, single-user, multi-device AI workflow environment. For the first milestone, the scope is intentionally limited to read-only custom pages.

## Goals

- Add a safe foundation for project-specific workbench pages.
- Let AI generate page definitions without giving it arbitrary code execution.
- Keep page definitions reviewable, diffable, and portable with the project.
- Prepare a clean integration path for future automation workflows and SYNC inbox data.
- Follow an open-source-friendly design: clear scope, explicit non-goals, testable milestones, and reviewable documentation.

## Non-Goals

This milestone does not implement:

- In-page file editing.
- Arbitrary React, JavaScript, or plugin execution generated by AI.
- Workflow execution buttons that run commands or mutate project state.
- Real mobile notification capture.
- Real SYNC network integration.
- Multi-user permissions or server-side tenancy.
- Full low-code visual workflow editing.

## User Stories

### Project Owner

As a user, I want each project to have a custom overview page so that I can see the information that matters for that project without opening multiple tools.

### AI-Assisted Project Setup

As a user, I want an AI agent to propose a project page schema so that I can quickly create a useful project dashboard while still reviewing the generated file before enabling it.

### Code Project

As a user working on a code project, I want a page that shows README content, Git status, recent agent summaries, and build-related notes.

### IP Design Project

As a user working on an IP design project, I want a page that shows worldbuilding notes, character document indexes, pending material inbox entries, and recent publishing summaries.

### Future SYNC Request

As a user, I want a request forwarded by SYNC to appear in the related project page so that I can decide what should happen next from the project context.

## Product Concept

The feature introduces Project Workbench Pages.

A Project Workbench Page is a read-only project-specific UI defined by a JSON schema and rendered by PRES. It is not a plugin and does not execute page-provided code.

Recommended storage for the first milestone:

```text
<project-root>/.mutsumi/pages/*.json
```

Rationale:

- The page belongs to the project.
- The definition can be reviewed in Git.
- AI-generated page changes can be presented as diffs.
- Future cross-device sync can treat page schemas as project files.

PRES-global page overrides can be considered later, but should not be part of the first implementation.

## Page Model

Each page definition should include:

- Stable page id.
- Display title.
- Optional description.
- Layout type.
- Data source declarations.
- Block list.
- Version field for schema migrations.

Example:

```json
{
  "schemaVersion": 1,
  "id": "project-overview",
  "title": "Project Overview",
  "description": "Read-only project status and context.",
  "layout": {
    "type": "twoColumn",
    "leftWidth": 0.58
  },
  "dataSources": {
    "project": { "type": "project" },
    "git": { "type": "gitStatus" },
    "readme": { "type": "markdownFile", "path": "README.md" },
    "agent": { "type": "agentSummaries", "limit": 5 }
  },
  "blocks": [
    {
      "type": "markdown",
      "title": "README",
      "source": "readme"
    },
    {
      "type": "statList",
      "title": "Git Status",
      "source": "git"
    },
    {
      "type": "timeline",
      "title": "Recent Agent Summaries",
      "source": "agent"
    }
  ]
}
```

## Supported Data Sources for Milestone 1

### `project`

Returns current PRES project metadata:

- id
- name
- path
- group id
- editor presets

### `gitStatus`

Returns the current project Git status using existing PRES Git command infrastructure.

The first version may expose a simplified shape:

- branch
- clean or dirty
- changed file count
- raw status text

### `fileList`

Returns a bounded list of files under a configured relative path.

Required constraints:

- Relative paths only.
- No parent traversal.
- Configurable max depth.
- Configurable max item count.
- Hidden files excluded by default.

### `markdownFile`

Reads and renders one Markdown file.

Required constraints:

- Relative path only.
- No parent traversal.
- File size limit.
- Rendered with sanitized output.

### `agentSummaries`

Returns project-scoped AI conversation summaries.

For Milestone 1, this may use a local mock or local store shape. It should be designed so real SYNC summary ingestion can be added later without changing page schemas.

## Planned Future Data Sources

These should be named in the schema design but not fully implemented in Milestone 1:

- `syncInbox`: requests forwarded by Mutsumi's-SYNC.
- `workflowRuns`: read-only automation execution history.
- `projectMemory`: project-scoped memory excerpts.
- `releaseHistory`: local or remote publishing records.

## Block Types for Milestone 1

### `statList`

Displays labeled values from a data source.

### `markdown`

Displays sanitized Markdown from a `markdownFile` source.

### `fileList`

Displays files from a `fileList` source.

### `table`

Displays simple row data.

### `timeline`

Displays ordered events such as agent summaries or future workflow runs.

### `callout`

Displays static page guidance or generated notes.

### `actionPlaceholder`

Displays a disabled or non-mutating placeholder for future workflow buttons.

Rules:

- `actionPlaceholder` must not execute commands in Milestone 1.
- It may open a details view, show explanatory text, or mark where a future workflow will attach.

## Rendering Requirements

- Pages must render inside the existing project detail area.
- Page rendering must be deterministic from schema plus data sources.
- Invalid page schemas must fail gracefully with a readable validation error.
- A missing data source must fail only the affected block, not the whole page.
- Blocks must use existing visual language: dense, restrained, rectangular, and project-tool oriented.
- Blocks must not appear as marketing cards or landing-page sections.
- Long text and file paths must not overflow their containers.

## Security and Safety Requirements

- Page schemas must not execute JavaScript.
- Page schemas must not define raw HTML.
- File reads must be restricted to the selected project root.
- Paths must be normalized and rejected if they escape the project root.
- Markdown rendering must sanitize unsafe HTML.
- Data source limits must prevent large accidental reads.
- Future AI-generated schemas must be validated before being saved or enabled.
- Future mutating actions must require explicit permissions and user confirmation.

## AI Generation Requirements

AI agents may generate Project Workbench Page schemas, but the output must be treated as a draft.

Draft flow:

1. Agent proposes a schema file.
2. PRES validates the schema.
3. PRES previews the page.
4. User reviews the schema diff and preview.
5. User enables or rejects the page.

Milestone 1 may implement only manual file placement and validation. The draft-review UI can be added later.

## Relationship to Automation Workflows

Automation workflows are intentionally out of scope for this milestone, but the page model must leave space for them.

Future workflow model:

```text
trigger -> context -> agent call -> draft output -> user approval -> action -> summary -> report
```

Future workflow node categories:

- trigger
- context
- agent
- action
- approval
- report

The page system should prepare for:

- Displaying workflow run history.
- Showing pending approvals.
- Hosting future workflow buttons.
- Showing agent-generated diffs before applying changes.

## Relationship to Mutsumi's-SYNC

SYNC integration is not required in Milestone 1.

The page system should prepare for:

- Showing read-only SYNC inbox requests.
- Showing globally injected memory excerpts.
- Showing summarized conversation records reported back to SYNC.

PRES should remain responsible for local project interpretation and execution.

## Implementation Plan

### Phase 0: Documentation and Schema Agreement

- Add this requirements document.
- Review naming, storage location, and non-goals.
- Open discussion issues for unresolved schema choices.

Acceptance criteria:

- The requirements document is merged.
- The first milestone remains read-only.
- Storage location and schema versioning are agreed.

### Phase 1: Schema Types and Validation

Create TypeScript types for page schemas and a validator.

Expected files:

- `src/lib/workbench/pageSchema.ts`
- `src/lib/workbench/pageValidation.ts`
- `src/lib/workbench/__tests__/pageValidation.test.ts`

Acceptance criteria:

- Valid example schemas pass.
- Unknown block types fail.
- Unknown data source types fail.
- Parent traversal paths fail.
- Missing required fields fail with readable messages.

### Phase 2: Project Page Discovery

Read `.mutsumi/pages/*.json` from a project root through Rust commands and frontend wrappers.

Expected files:

- `src-tauri/src/commands/workbench.rs`
- `src-tauri/src/commands/mod.rs`
- `src/lib/tauri.ts`
- `src/lib/workbench/pageLoader.ts`

Acceptance criteria:

- Project pages are discovered from the selected project.
- Invalid JSON is reported without crashing.
- Missing `.mutsumi/pages/` returns an empty page list.
- Path traversal is rejected.

### Phase 3: Data Source Resolution

Implement read-only data source resolvers.

Expected files:

- `src/lib/workbench/dataSources.ts`
- `src-tauri/src/commands/workbench.rs`
- `src/lib/workbench/__tests__/dataSources.test.ts`

Acceptance criteria:

- `project` resolves from current store state.
- `gitStatus` uses existing Git status command behavior.
- `fileList` respects depth and count limits.
- `markdownFile` respects file size and path safety.
- `agentSummaries` can return mock or locally stored summaries.

### Phase 4: Page Renderer

Render validated schemas in the project detail view.

Expected files:

- `src/components/workbench/WorkbenchPageView.tsx`
- `src/components/workbench/blocks/StatListBlock.tsx`
- `src/components/workbench/blocks/MarkdownBlock.tsx`
- `src/components/workbench/blocks/FileListBlock.tsx`
- `src/components/workbench/blocks/TableBlock.tsx`
- `src/components/workbench/blocks/TimelineBlock.tsx`
- `src/components/workbench/blocks/CalloutBlock.tsx`
- `src/components/workbench/blocks/ActionPlaceholderBlock.tsx`
- `src/components/ProjectDetail.tsx`

Acceptance criteria:

- A project with one valid page can display it.
- A project with multiple pages can switch between them.
- Block-level errors do not crash the whole page.
- Empty states are clear and non-destructive.
- Layout remains usable at the existing app minimum width.

### Phase 5: Example Pages and Documentation

Add example page schemas and user-facing documentation.

Expected files:

- `docs/examples/workbench/project-overview.json`
- `docs/examples/workbench/ip-design-overview.json`
- `docs/workbench-pages.md`

Acceptance criteria:

- Example schemas validate.
- Documentation explains where to put page files.
- Documentation explains supported data sources and block types.
- Documentation states that Milestone 1 is read-only.

## Testing Strategy

- Unit test schema validation.
- Unit test path normalization and traversal rejection.
- Unit test data source resolver behavior where possible.
- Component test renderer behavior for supported block types if the current test setup supports React rendering.
- Manual test with a real project containing `.mutsumi/pages/project-overview.json`.
- Run `npm run build`.
- Run `cargo check --manifest-path src-tauri/Cargo.toml` after Rust command changes.

## Open Questions

- Should page definitions support project-local files only, or also PRES-global personal pages?
- Should Markdown allow a small safe subset of HTML, or strip all raw HTML?
- Should `agentSummaries` be stored in the existing JSON store or a separate project-scoped file?
- Should example schemas be generated by PRES in new projects, or only documented?
- What is the first real SYNC request shape PRES should display when SYNC integration begins?

## Community Contribution Guidelines

Contributors should:

- Keep the first milestone read-only.
- Avoid adding arbitrary script execution to page schemas.
- Add tests for every schema or path-safety rule.
- Keep new UI consistent with the existing desktop app style.
- Prefer small PRs by phase.
- Document any schema changes and bump `schemaVersion` only when necessary.

Suggested PR sequence:

1. Schema and validation.
2. Project page discovery.
3. Read-only data sources.
4. Renderer and project detail integration.
5. Examples and user documentation.

## Definition of Done for Milestone 1

- A project can contain `.mutsumi/pages/*.json`.
- PRES can discover and validate those pages.
- PRES can render at least one selected page in the project detail area.
- Supported data sources are read-only.
- Invalid schemas produce readable errors.
- File access cannot escape the project root.
- Example pages and user documentation are included.
- Build and Rust checks pass in CI.
